面向对象

发表于 2017-04-22 | 分类于 Program

大家可能会觉得很奇怪：

为什么用接口？

既然可以用一个类做到，为什么写接口？这里会用一个新系列，介绍相关问题。

首先，引入第一个概念：粒度。

粒度

粒度是程序的组合单元规模。以在线商城为例，第一种组合方式是，用户模块、管理员模块。第二种，登陆、注册、查询用户、查询订单 · · · · 这就是不一样的粒度。

从粒度出发，重新分析 OO 和 PO 就会发现：

OO（面向对象）提高了编程的粒度。

PO（面向过程）

PO 的基本单元是函数，是以事件为中心的编程思想，关注的是解决问题的步骤。一个 PO 的程序员，他会分析问题的解决步骤，然后用一个个函数实现。

这适合以前的小型程序，但面对大中程序，弊端就出现了，函数的粒度太低。上面的在线商城，如果用第二种方式，那么就可能有 1000 个函数，项目的维护、分析要花掉大量资源。

为了适应这种情况，另一种设计思想 OO 诞生了。

OO（面向对象）

OO 的基本单元是类，是以对象为中心的思想，关注的是构成问题的事物。类封装了过程，以及相关的函数、数据，粒度提升了一个层次。这个阶段，OO 程序员就要分析事物间的关联，封装成对象。这种能力叫做 抽象。从此，程序员又多了一道门槛。

OO 的出现，项目开始变得更有层次。在线商城的第一种组合，可以分为用户、管理员模块。理论上，每个模块还能无限细分下去。可以说，OO 影响了软件分工。哪一天，可能会出现 “ 按钮工程师 ”，前提是 Web 没有被淘汰。

结语

从 粒度 的角度分析，PO 迭代到 OO 无疑减轻了程序员的工作量。同时，互联网产品也有了大规模爆发的工程基础。
